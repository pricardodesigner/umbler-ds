Paleta — 8 cores

O botão X aparece no hover em position: absolute — sem empurrar a largura da pill nem reposicionar as etiquetas adjacentes.

Sem ícone

Design Frontend Vendas Urgente

Best Practices

  • Use tags para categorização de entidades (contatos, conversas, leads) — não para status.
  • Para status semântico (erro/sucesso/aviso), prefira Badges com bg-* ou Alertas.
  • Consistência de cor por significado: trate Volcano = Lead quente, Green = Cliente ativo, etc. como convenção fixa na sua organização.
  • O X é injetado pelo JS window.umbInitTags() — opt-out com data-no-close="true" (ex.: tags dentro de popover de seleção onde o controle é por checkbox).
  • Para remoção programática, escute o click no .umb-tag-close e remova o elemento — o DS já faz isso quando não há lógica externa.
  • Nunca use mais de 3–4 cores diferentes na mesma tela — vira ruído visual.

<!-- Tag com ícone + X injetado automaticamente -->
<span class="umb-tag tag-geekblue">
  <i class="ph ph-user-circle"></i>Campanha VIPs
</span>

<!-- Sem X (opt-out) -->
<span class="umb-tag tag-green" data-no-close="true">Cliente ativo</span>

<!-- Init (auto-chamado no DS, necessário em apps terceiros) -->
<script>window.umbInitTags();</script>

Tag Popover

UmbTagPopover — seleção multi-item dentro de dropdown. Busca, segmented opcional, seção "Selecionadas" dinâmica.

Exemplo interativo

Etiquetas
Todas
Campanha VIPs
Lead quente
Cliente ativo
Parceiro
Trial
Urgente

Clique no botão à esquerda pra abrir o popover. Marque/desmarque em qualquer ponto da linha — o checkbox acompanha. A seção Selecionadas (N) aparece automaticamente no topo com os items marcados e some quando nenhum está selecionado. Desmarcar um clone ali em cima também desmarca o original na lista "Todas" abaixo.

Best Practices

  • Trigger deve ter data-bs-auto-close="outside" — impede que cliques nos itens fechem o popover. Sem isso, o Bootstrap fecha ao primeiro clique e o usuário não consegue marcar múltiplos items.
  • O botão X do header usa aria-label="Fechar" — o handler delegado do DS fecha o dropdown manualmente quando detecta esse botão.
  • Clique em qualquer ponto de .umb-tag-pop-item toggle o checkbox. Isso é init automático pelo DS (delegação no document).
  • Seção "Selecionadas (N)": basta colocar um slot vazio <div class="umb-tag-pop-selected"></div> antes da lista "Todas". O init (umbInitTagPopover) popula automaticamente com label + clones dos items marcados e esconde quando N === 0. Clicar/desmarcar em qualquer lado (seção ou lista) mantém os dois em sincronia — não renderize manualmente.
  • Ordenação: .umb-tag-pop-search (flex row, gap 8px) agrupa o input-group de busca + o trigger .umb-tag-pop-sort-trigger (btn-icon btn-text, fora do input pra evitar cortar com o overflow:hidden). O menu .umb-tag-pop-sort-menu é popup custom (não Bootstrap dropdown — conflitava com o dropdown-menu pai do popover) posicionado absolutamente à direita. Opções data-umb-sort="default" (data-umb-order, congelado no primeiro render) e data-umb-sort="alpha" (localeCompare pt-BR de .umb-tag). Só use duas opções — mais que isso vira filtro, não ordenação.
  • Use data-no-close="true" nas tags dentro do popover — evita que o helper de X das etiquetas injete botão de remover em tags que são apenas visual.
  • Largura fixa 320px. Não altere — foi calibrada para 6-8 itens sem scroll excessivo em desktop.

Best Practices

  • Footer sem border-top. A separação vem do margin-top: 32px. Diferente de .offcanvas-footer. Em mobile, vira margin-top: 24px.
  • Hierarquia Voltar/Avançar é fixa: Voltar sempre btn-text btn-lg com ph-caret-left; Avançar sempre btn-primary btn-lg com ph-caret-right. Essa é a única situação do DS onde btn-primary sólido é padrão recorrente — exceção explícita à §1.
  • Primeira tela sem steps. Quando a primeira tela é uma escolha de origem/tipo que determina o fluxo (ex: "enviar para contatos do Talk" vs "importar arquivo"), esconda o .steps-container. Steps aparecem a partir do 2º passo, renderizados conforme o caminho.
  • Steps dinâmicos e variáveis. O número e nomes dos passos devem mudar conforme a escolha inicial. Render via JS a partir de uma lista de passos do caminho escolhido — evite markup duplicado.
  • Sub-views dentro de um passo. Se um passo tem uma tela informativa inicial + uma tela detalhada opcional (ex: resumo → tabela), use [data-subview] no mesmo .umb-wiz-screen. Não crie um step extra para isso.
  • Botão fechar no canto superior direito. .umb-wiz-close com position: absolute; top:20px; right:20px. Chama um confirm se o usuário tiver progresso não salvo.
  • Responsivo automático. Abaixo de 768px: padding reduzido, título menor, footer empilha vertical com Avançar ocupando largura total. Sem overrides manuais na maioria dos casos.

Best Practices

  • Dois estados, uma posição. Dropzone (vazio) e File Pill (escolhido) ocupam o mesmo lugar da tela — alterne via display. Não deixe os dois visíveis simultaneamente.
  • Hover = dragover. A mesma regra CSS (borda primary + bg hover) dispara tanto no :hover quanto em .is-dragover. Adicione/remova a classe via JS durante os eventos dragover/dragleave.
  • Ícone do arquivo por tipo. Use ph-file-csv, ph-file-pdf, ph-image, ph-file-doc conforme o tipo detectado. Mantém a categoria visível antes do nome.
  • Metadata relevante no meta. Linhas/tamanho/encoding para CSV; dimensões/tamanho para imagem; páginas/tamanho para PDF. Não polua — 2–3 campos máximo, separados por ·.
  • Botão remover sempre presente. File Pill com .btn-icon.btn-text.btn-lg + ph-trash à direita. Remove o arquivo e volta pro estado Dropzone.
  • Pré-visualização fora do pill. Para mostrar as primeiras linhas do CSV ou preview da imagem, posicione abaixo do File Pill, com borda de 1px em --umb-border e radius 12px. Sem fundo colorido.
  • Limites visíveis. Os chips embaixo do Dropzone (.CSV, .VCF, Até 10MB) comunicam aceites e limite. Não esconda — reduzem a ansiedade do usuário antes do upload.

Best Practices

  • Cor padrão do ícone: neutral. O default de .umb-path-icon é var(--umb-text-mid). Não adicione classe modificadora (.primary, .success, .danger, .warning) a menos que haja destaque semântico claro.
  • Regra dos 5 — cores nos ícones só em listas curtas. Em listas com até 5 opções, modificadores de cor podem destacar a semântica de cada card (.primary para ação recomendada, .success para aprovação, etc). Em listas com 6+ opções, todos os ícones devem ficar neutros — muitas cores viram ruído visual e destroem a hierarquia.
  • Ideal 3 opções, máximo 5 antes de virar lista. Cards de decisão têm peso visual — cada um ocupa espaço considerável na tela. Com 6+ opções, considere trocar por Radio ou Select (componente adequado pra listas longas de escolha única).
  • Badge "Recomendado" em no máximo 1 card. Se tudo é recomendado, nada é. Use .umb-path-card.recommended no card mais provável/seguro.
  • Caret-right no fim. Sempre presente — comunica "clique pra seguir". Pode ser omitido em cards que disparam ações inline (sem navegação), mas nesses casos geralmente Path Card não é o componente certo.
  • Cor do ícone comunica semântica. Quando usar modificador: .primary para ação padrão/ativa, .success para confirmação/aprovação, .warning para aviso. Evite .danger — remoções destrutivas merecem confirmação modal, não card.
  • Subtítulo é obrigatório. O título sozinho não carrega contexto suficiente. Sempre explique em 1 linha o que acontece ao clicar.
  • Paridade de comprimento entre cards. No desktop, título e subtítulo de cada card devem ocupar no máximo 1 linha — se não couber em max-width: 640px, reescreva mais curto. No mobile, todos os cards da mesma lista devem ter a mesma quantidade de linhas em título e subtítulo (se um quebra pra 2 linhas, todos quebram). Texto mais longo cria falsa percepção de importância e distorce a decisão — a paridade mantém o peso visual equilibrado entre opções paralelas.

Composer mobile — estados do botão primário

No mobile, o botão primário do .umb-chat-composer-toolbar-right alterna conforme o conteúdo do textarea. Quando vazio, o botão é o microfone (gravar áudio). Ao digitar, o microfone é substituído pelo paper-plane (enviar). Só um dos dois existe por vez — o desktop mantém o microfone fixo porque conta com Enter do teclado para enviar.

1. Vazio — botão de gravar áudio

Paulo Ricardo

Sem texto no textarea: o botão primário é o microfone. Tocar inicia a gravação de áudio.

2. Digitando — botão de enviar

Paulo Ricardo

Com texto digitado: o microfone é substituído pelo paper-plane. Tocar envia a mensagem.

Regra de uso

No mobile, o componente precisa reagir ao input do textarea: value.length === 0 renderiza microfone, value.length > 0 renderiza paper-plane. Nunca mostre os dois ao mesmo tempo — isso introduz decisão desnecessária. No desktop, mantenha o microfone fixo: Enter é o atalho canônico de envio.
